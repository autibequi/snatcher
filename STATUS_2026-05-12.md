## O que precisa de validação manual / ação operacional

**Antes de subir em prod**:

1. **Rodar migrations** no DB de dev/staging primeiro:
   ```bash
   cd backend-go
   go mod tidy   # baixar golang-migrate v4
   make migrate-up
   ```
   Esperado: ~116 migrations aplicadas sem erro. Se DB já tinha as 76 antigas trackeadas em `schema_migrations` legacy → fazer `migrate force <max>` antes (ver `docs/migrations.md`).

2. **Validar `get_param`**:
   ```sql
   SELECT get_param('quality_threshold','global',NULL);  -- deve retornar 0.40
   SELECT COUNT(*) FROM llm_autonomy;                    -- 9
   SELECT COUNT(*) FROM modems;                          -- 3
   ```

3. **Cadastrar `accounts` com afinidade fixa modem**:
   - Cadastrar cada conta WA Business em `accounts` com `modem_id` correto
   - SQL exemplo: `INSERT INTO accounts (phone, modem_id, status, daily_send_quota) VALUES ('+5511...', 1, 'warming', 20);`
   - Cadastrar `group_admins` mapeando cada grupo → contas autorizadas

4. **Ativar flag de senders novos** (depois de validar 1d em dev):
   ```sql
   UPDATE tunable_parameters SET current_value = 1 WHERE param_name = 'use_send_queue' AND scope_type = 'global';
   ```
   O dispatcher antigo entra em no-op automaticamente.

5. **Ativar Algo tick novo** (após validar `catalog` populado):
   ```sql
   UPDATE tunable_parameters SET current_value = 1 WHERE param_name = 'use_algo_tick' AND scope_type = 'global';
   ```

6. **Fold catalog**: disparar UMA vez pra popular `catalog` a partir de `catalogvariant`:
   ```bash
   curl -X POST http://admin.snatcher/api/admin/fold-catalog -H "Authorization: Bearer ..."
   ```
   Aguarda 1-2 dias → `catalog` populado → flip `catalog_source=1`.

7. **Criar 2 grupos WA dedicados** (Fase 6):

## Gotchas conhecidos

1. **`go mod tidy` precisa rodar** pra baixar `github.com/golang-migrate/migrate/v4` antes do primeiro build. O ambiente da sessão noturna não tinha internet, então `go.sum` ficou desatualizado.

2. **BrokenImports no LSP** que apareceram nesta sessão são **falsos positivos** — LSP estava olhando arquivos isolados sem workspace válido. Memory `defensive/cache_repos_no_git.md` documenta isso. `go vet` real (com `go mod tidy` rodado) deve passar limpo na maioria.

3. **Repo `.cache/snatcher/.git` é stub vazio** — nenhum commit foi feito. Os arquivos foram escritos diretamente. Pedrinho precisa criar branch no repo real (clone do GitHub) e copiar mudanças.

4. **Algumas funções têm stubs honestos**:
   - `sendViaEvolution` no sender (precisa integrar com `internal/messaging/evolution/`)
   - `RunSentimentAnalyze` (depende de Evolution chat-history API)
   - `PromoBotWebhookHandler` (depende de LLM call real pra Q&A)
   - `LLM call` nos loops L5-L9 (lógica heurística por ora; LLM real é Fase 5.5/futura)

5. **L9 `content_optimize` SÓ ativa** quando: (a) Pedrinho muda status manualmente para `active`; (b) há ≥500 conversões nos últimos 60 dias. Gate verificado em runtime.

6. **`fold_catalog` é one-shot** — disparar 1× quando houver dados em `catalogvariant`. Idempotente via UNIQUE em `dedup_key`.

7. **Dual-write controlado por flags**:
   - `use_send_queue` (default 0): dispatcher antigo ativo
   - `use_algo_tick` (default 0): AutoMatch antigo ativo
   - `catalog_source` (default 0=legacy): Algo lê de `catalogvariant`
   
   Cutover gradual: dev → flag → 24h → prod → flag → 7d → deprecate o antigo.
